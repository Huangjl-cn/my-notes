四个不接受 Checked Exception 的函数式接口

Java 的函数式编程建立在四个核心接口之上，它们的设计目标之一就是在 Lambda 表达式中简洁使用。如果允许 throws checked exception，每个 Lambda 都要处理或重新抛出，破坏函数式风格的流畅性。

关键设计取舍：如果允许 throws checked exception，那么每个 Lambda 都要显式处理，破坏链式调用的流畅性。实际开发中遇到 IOException 等受检异常，在 Lambda 内用 try-catch 包装成 RuntimeException，或使用社区库的 ThrowingFunction 等变体接口。

与 CompletableFuture 的关系

CompletableFuture 的方法大量依赖这四个函数式接口，它们构成了异步编排的语法基础：

方法参数类型作用
supplyAsync(Supplier<U>)Supplier异步执行有返回值的任务
runAsync(Runnable)Runnable异步执行无返回值的任务
thenApply(Function<T,R>)Function对上一个异步结果同步转换
thenAccept(Consumer<T>)Consumer消费上一个异步结果，无返回值
thenCompose(Function<T,CompletionStage>)Function串联异步任务
thenCombine(CompletionStage, BiFunction)BiFunction合并两个异步结果
为什么这些接口在 CompletableFuture 中也不允许受检异常？CompletableFuture 的设计理念是将异步任务的异常包装在 Future 对象中，通过 join() / get() / exceptionally() 统一处理。如果 Lambda 可直接抛出受检异常，内部捕获后仍需包装成 CompletionException（非受检）。不如要求用户在 Lambda 内自己处理，保持接口签名干净。

CompletableFuture 解决了什么问题

在 CompletableFuture 出现之前，Java 的异步编程面临两个方向的痛点：原始的 Future 太弱，手动 Thread 太原始。

相比原始 Future：

原始 Future 问题CompletableFuture 解决方案
get() 阻塞线程，浪费资源通过回调（thenApply/thenAccept）非阻塞链式处理
难以组合多个异步任务thenCombine、thenCompose、allOf、anyOf
异常处理只能靠 ExecutionExceptionexceptionally、handle 提供声明式恢复
无法手动完成一个 Futurecomplete / completeExceptionally 可外部设置结果
超时处理繁琐orTimeout、completeOnTimeout 直接支持

相比手动 Thread + Runnable：

手动 Thread 痛点CompletableFuture 改进
run() 无返回值，需要共享变量 + 同步supplyAsync 直接返回 Future<T>
子线程异常只能打印，主线程无法感知异常捕获存入 Future，join 可获取
任务编排需手写 CountDownLatch、wait/notify声明式 thenCombine、allOf
线程管理需自己创建或管理线程池内置 ForkJoinPool.commonPool() 或自定义 Executor
原始 Future（阻塞两次）vs CompletableFuture（非阻塞链）
// 原始 Future — 需要两次阻塞 Future<String> f1 = executor.submit(() -> fetchUser()); String user = f1.get(); // 阻塞！ Future<Integer> f2 = executor.submit(() -> calcScore(user)); Integer score = f2.get(); // 再次阻塞！ // CompletableFuture — 非阻塞链式 CompletableFuture.supplyAsync(() -> fetchUser()) .thenApply(user -> calcScore(user)) .thenAccept(score -> System.out.println(score));

原始并发：Java 1.0 ~ 1.4 的时代

在 Java 5 之前，只有 java.lang.Thread 和 java.lang.Runnable。同步靠 synchronized、wait() / notify()。

原始并发的痛点
• 每个任务创建一个 OS 线程，创建/销毁开销大
• 线程数不可控，几千线程就 OOM
• Runnable.run() 返回 void，无返回值
• 异常无法跨线程传递
• 难以取消、超时、等待多个任务
ExecutorService 的改进（Java 5）
• 线程复用，减少创建/销毁开销
• 控制最大并发数，防止资源耗尽
• 内置任务队列，平滑处理突发流量
• 支持返回值（Callable + Future）
• 异常传递、批量等待（invokeAll）

在没有 Callable 的年代，如何获取返回值

Java 5 之前要实现有返回值的并发任务，必须手动用共享变量 + 同步机制模拟。

方法一：wait/notify + 共享对象
class ResultHolder { private Integer value; private boolean done = false; public synchronized void set(Integer v) { this.value = v; this.done = true; notify(); // 唤醒等待的主线程 } public synchronized Integer get() throws InterruptedException { while (!done) { wait(); } return value; } }
方法二：join + 静态变量（更简单但同样脆弱）
private static Integer result; Thread t = new Thread(() -> { int sum = 0; for (int i = 0; i < 100; i++) sum += i; result = sum; }); t.start(); t.join(); // 阻塞等待线程死亡 System.out.println(result);
这些原始方式的共同缺陷：大量样板代码、容易出错（忘记同步、信号丢失）、无法传递子线程异常、不支持超时、无法取消、无法组合多个任务。这正是 Java 5 引入 Callable<V> 和 FutureTask 的原因。

Callable：第一个有返回值的并发抽象

Callable<V> 是 Java 5 引入的，它不属于"原始并发"（Java 1.0~1.4）。整个演进可以分为三个阶段：

阶段代表性技术是否有返回值异常处理
① 原始并发 (Java 1.0-1.4)Thread, Runnable, wait/notify❌ 无子线程异常无法传递
② Java 5 改进Callable<V>, Future, ExecutorService✅ 有Future.get() 抛出 ExecutionException
③ Java 8+ 函数式异步CompletableFuture, CompletionStage✅ 有链式异常恢复 (exceptionally, handle)
Callable 使用示例
ExecutorService executor = Executors.newSingleThreadExecutor(); Future<Integer> future = executor.submit(new Callable<Integer>() { // 注意：call() 允许 throws Exception public Integer call() throws Exception { Thread.sleep(1000); return 42; } }); Integer result = future.get(); // 阻塞，但比手搓 wait/notify 简洁得多 executor.shutdown();
Callable 相比原始方式进步很大，但它仍然有阻塞的 get()，且难以组合多个异步任务。CompletableFuture 在它之上进一步提供了非阻塞和丰富编排能力。

虚拟线程：让阻塞不再昂贵

虚拟线程本质是 JVM 管理的用户态线程，创建成本极低（几百字节），百万级并发轻松实现。遇到阻塞自动让出载体线程，不浪费系统资源。

虚拟线程的两种创建方式
// 方式 1：直接启动 Thread.startVirtualThread(() -> System.out.println("Hello")); // 方式 2：通过 ExecutorService（推荐，支持 try-with-resources） try (var executor = Executors.newVirtualThreadPerTaskExecutor()) { executor.submit(() -> System.out.println("Hello")); }
结构化并发示例
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) { Future<String> user = scope.fork(() -> fetchUser()); Future<Integer> order = scope.fork(() -> fetchOrder()); scope.join(); // 等待所有子任务 scope.throwIfFailed(); // 如果有异常则抛出 return new Response(user.resultNow(), order.resultNow()); } // 作用域结束，自动取消未完成的子任务
作用域值：ThreadLocal 的现代替代
private static final ScopedValue<String> USER = ScopedValue.newInstance(); ScopedValue.where(USER, "Alice").run(() -> { System.out.println("Hello " + USER.get()); // Hello Alice }); // 离开作用域后，USER 不可访问

CompletableFuture 还需要吗？

虚拟线程的兴起并不意味着 CompletableFuture 的消亡——它们解决不同的问题。

角度虚拟线程CompletableFuture
核心价值让阻塞不再昂贵，支持海量并发声明式异步编排，组合复杂依赖
适合场景简单高并发 I/O（每个任务独立）任务有依赖、需聚合、竞速、超时、恢复
代码风格同步顺序代码，易读易调试链式回调，适合函数式编程
典型例子Web 服务器接收请求，转发给下游电商下单：库存、价格、优惠并行调用后聚合
最佳实践：结合两者。用虚拟线程运行 CompletableFuture 的任务，既享受编排能力，又享受轻量调度。
虚拟线程 + CompletableFuture 组合
ExecutorService vtExec = Executors.newVirtualThreadPerTaskExecutor(); CompletableFuture.supplyAsync(() -> fetchUser(), vtExec) .thenApply(user -> fetchOrders(user)) .thenAccept(orders -> System.out.println(orders));

虚拟线程在 CPU 密集型任务上还有优势吗？

CPU 密集型 × 虚拟线程
−5%~15%
虚拟线程调度器（ForkJoinPool）的工作窃取和上下文切换有额外开销。纯计算任务下吞吐量反而不如固定大小平台线程池。
I/O 密集型 ✓ 虚拟线程
+400%
等待时间远大于计算时间，虚拟线程可以在阻塞时让出载体线程去执行其他任务，利用率极高。
"轻量"指的是内存占用和创建成本，不是计算速度。对于 CPU 密集型任务，你根本不需要百万线程，只需要等于核心数的线程。此时虚拟线程的轻量特性毫无用处，反而因调度器额外开销拖慢速度。

推荐做法：CPU 密集型 → Executors.newFixedThreadPool(N_CPUS)；I/O 密集型 → 虚拟线程；混合任务 → 分别使用不同的执行器，组合结果。

M:N 调度 + Continuation

虚拟线程的核心原理是 M:N 调度（M 个虚拟线程运行在 N 个平台线程上，N 通常等于 CPU 核心数）和 Continuation（续体）——一个可暂停、可恢复的执行单元，保存调用栈、局部变量、程序计数器等状态，存储在 JVM 堆中而非内核栈上。

关键概念澄清

载体线程 = Java 平台线程 = 操作系统线程
真正工作（执行 CPU 指令）的永远是载体线程。虚拟线程本身是一套指令和状态，它自己不会执行任何 CPU 指令。虚拟线程的作用不是亲自干活，而是让载体线程能够高效地"穿梭"于大量任务之间，避免因任务阻塞而浪费载体线程。
虚拟线程是一次性的吗？
是的。每个虚拟线程只能被启动一次（与平台线程一致），run() 方法执行完毕或抛异常后状态变为 TERMINATED。复用的是载体线程——底层平台线程被反复用来挂载不同的虚拟线程。由于虚拟线程创建成本极低（微秒级别），可以放心大量创建。
虚拟线程 = 一段可挂载/可卸载的执行状态的抽象
虚拟线程保存的不是载体线程的栈内容，而是自己本该运行在载体线程上的那部分栈帧。它是一个包含完整执行上下文（栈帧、PC、局部变量等）的容器，可以被附着到任意一个载体线程上运行，并在阻塞时脱离。可以类比为：载体线程 = CPU 核心，虚拟线程 = 进程的镜像快照。

ForkJoinPool：两个池子的隔离

虚拟线程调度器使用的是一个独立的 ForkJoinPool 实例，与 ForkJoinPool.commonPool() 完全隔离。

特性ForkJoinPool.commonPool()虚拟线程调度器
获取方式静态方法直接获取不可直接获取（JDK 内部创建）
用途并行流、CompletableFuture（默认）作为虚拟线程的载体线程池
线程数availableProcessors() − 1默认 availableProcessors()，可通过系统属性调整
工作窃取模式LIFO（后进先出，适合分治任务）FIFO（先进先出，适合大量短暂任务）
用户任务提交可以（commonPool().submit()）不可以（用户代码无法直接访问）
为什么要有两个池子？隔离关注点——并行流任务和虚拟线程任务不互相干扰；独立调优——可以单独调整虚拟线程调度器参数；安全性——防止用户代码意外破坏虚拟线程调度器内部状态。

为什么都叫 ForkJoinPool？ForkJoinPool 是一个通用工作窃取线程池实现，其工作窃取特性也适合虚拟线程调度。Java 团队选择复用现有成熟组件，而非发明新类——就像 ArrayList 既可以做购物车也可以做学生名单，实例不同、用途不同。

调度器与执行器

虚拟线程只能在 ForkJoinPool 上运行吗？
默认情况下，虚拟线程的调度器是 JVM 内置的一个专用 ForkJoinPool。技术上可以通过系统属性或内部 API 指定其他调度器，但官方不建议普通用户这样做——默认调度器已经过充分优化，试图为虚拟线程自定义调度器通常是反模式。
Executors.newVirtualThreadPerTaskExecutor() 返回的是什么？
返回的是一个 ExecutorService（具体是 ThreadPerTaskExecutor 内部类实例）。每次 submit 都创建一个全新的虚拟线程来运行该任务。它本身不是一个虚拟线程，而是一个虚拟线程工厂 + 执行器。所有由它创建的虚拟线程共享同一个底层载体线程池。

关键要点

常见误区
• "Java 21 以后不需要 CompletableFuture"
• "虚拟线程让所有代码都更快"
• "虚拟线程 = 一个状态码"
• "ForkJoinPool.commonPool() 就是虚拟线程的池子"
• "虚拟线程可以复用（像线程池那样）"
正确理解
• CompletableFuture 做编排，虚拟线程做执行，职责不同
• 虚拟线程加速的是 I/O 密集型，CPU 密集型反而略慢
• 虚拟线程是完整的 java.lang.Thread 对象，有自己的栈帧
• 虚拟线程调度器是独立的 ForkJoinPool，与 commonPool 隔离
• 虚拟线程一次性的，复用发生在载体线程层面

一句话记忆法

函数式接口是异步编程的语法基础 → CompletableFuture 用它们实现了声明式任务编排 → 虚拟线程 解决了平台线程的阻塞浪费问题 → 两者结合 是 Java 25 时代的最佳实践。

记住这个公式：虚拟线程（执行引擎）+ CompletableFuture（编排工具）+ 结构化并发（生命周期管理）= 现代 Java 并发的完整工具箱。
