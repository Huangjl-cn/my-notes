直觉告诉我们：这就是三个缓存断点。但为什么要设置三个而不是一个？为什么 system prompt 要拆成两块分别放断点？这引出了第一个需要搞清楚的问题——API 到底按什么顺序处理这些东西？
API 的固定处理顺序

HTTP 请求体是一个 JSON 对象，字段在序列化层面没有顺序。但 Anthropic API 在组装 prompt 前缀时，有一个固定的处理顺序，不管 JSON 里各字段怎么排列。文档 ↗

prompt 前缀 = tools + system + messages（API 固定组装顺序，与 JSON 字段顺序无关）
tools 工具定义
system[0] 身份声明
system[1] 指令集
messages 对话历史
BP ① (cache)
system[0] 末尾
BP ② (cache)
system[1] 末尾
BP ③ (cache)
最后一条 assistant 末尾
prompt 的设计顺序，直接决定了缓存失效的级联关系。
级联失效规则

前缀匹配的逻辑：前面一变，后面全废，哪怕后面的内容一模一样。文档 ↗

你改了什么 哪些缓存会失效 影响范围
改 tools tools + system + 所有 messages 全部失效
改 system system + 所有 messages 部分失效
改 messages 仅 messages 部分 最小影响
为什么 system prompt 要拆成两块

把 system prompt 拆成两块分别放断点，是一个精妙的隔离策略。system[0] 是一句简短的模型身份声明（几乎永远不会改），system[1] 是详细的指令集（可能随 Claude Code 版本更新而变化）。

不拆分（一整块 system）
tools 定义
system = 身份声明 + 指令集（一整块）
← 唯一断点：指令集更新 →
tools 缓存也废了 ✗
messages
拆分（两块 system）
tools 定义
system[0]：身份声明
← 断点 ①：tools + sys[0] 缓存
system[1]：指令集
← 断点 ②：断点① + sys[1] 缓存
↑ 指令集更新 → 断点①仍命中！
messages
分开设置断点意味着：指令集更新了，身份声明和 tools 的缓存还能继续命中。这是一层精妙的隔离。
不是分段哈希，是累积前缀

看到三个 cache_control 出现在不同位置，很容易误以为它们是三段独立的缓存条目。实际上每个断点存的是从 prompt 开头到断点为止的完整前缀哈希。文档 ↗

1
Hash1 =
H( tools + system[0] )
2
Hash2 =
H( tools + system[0] + system[1] )
3
Hash3 =
H( tools + system[0] + system[1] + 全部 messages )
关键结论：Hash3 天然包含 Hash2 命中的范围，Hash2 天然包含 Hash1 命中的范围。 H3 天然包含前两者。它们不是并列关系，是嵌套关系。
嵌套结构可视化

每个断点都告诉 API："到这为止，给这整段区域生成一个缓存 key。"

tools { name: "search", input_schema: {...} }
system[0] "You are Claude Code, Anthropic's CLI..." 断点 ①
system[1] 完整指令（可长达几千 token） 断点 ②
messages user + assistant 对话历史 断点 ③
新消息 "那回滚呢？" ← 不缓存

断点 ① 覆盖范围最小，断点 ③ 覆盖从 tools 到最新消息的全部内容。新消息在断点之外，每次都需要重新计算。

误解 vs 实际
误解：分段哈希
三段各自独立存储
cache-A = H(tools+sys[0])
cache-B = H(sys[1] only)
cache-C = H(messages only)
改 sys[1] → B 失效，A、C 还在
但 A+C 没有 sys[1]，拼不回去
实际：累积前缀
每个断点从开头算起
cache-A = H(tools+sys[0])
cache-B = H(tools+sys[0]+sys[1])
cache-C = H(tools+...+messages)
改 sys[1] → B 失效
A 还能用，只重算 sys[1]+msgs
什么是 block

Block 是 prompt 的最小组成单元。你在 JSON 里看到的每一个 {"type": "text", "text": "..."} 就是一个 block，每一个 {"type": "tool_use", ...} 也是一个 block。一轮对话通常产生 2~5 个 block，不是固定的。

一轮 assistant 回复 ≈ 多个 block
{ "role": "assistant", "content": [ { "type": "text", "text": "让我搜索一下文档..." }, ← block 1 { "type": "tool_use", "name": "web_search", ... } ← block 2 ] }

block 数 ≠ 轮数，一轮对话可能产生多个 block

回溯窗口的工作原理

每次请求到达 API 时，从每个 cache_control 断点位置开始，API 往后退着找——计算当前位置的完整前缀哈希，去缓存库里查，找不到就退一个 block 重新算、重新查。最多退 20 步。文档 ↗

从断点位置向前回溯（退着找）：

[15] 命中! ← [16] [17] [18] [19] ··· [33] [34] [35]

退 20 步到 block[15] → 找到！这是 10 轮前写的缓存 → 命中！只重算 block[16]~[35]

回溯步骤
当前断点位置: block[35] → 算 hash(前缀+到35) → 缓存库没有 → 退1步，算 hash(前缀+到34) → 没有 → 退1步，算 hash(前缀+到33) → 没有 ... → 退20步，算 hash(前缀+到15) → 找到了！ → 命中！只重新算 block[16]~block[35] 如果退了 20 步都没找到： → 放弃。全量从头开始算。
正常对话：每轮都命中

因为每轮对话只新增 2~5 个 block，从当前断点到上一轮断点（上一次缓存写入的位置），距离远远小于 20。

追加式对话的缓存流
第1次请求: 断点③设在 msg1 末尾 → 写入 cache-A (覆盖 tools+sys+msg1) 第2次请求: 断点③设在 msg2 末尾 → API 发现 prefix 匹配 cache-A → 命中！只算 msg2 第3次请求: 断点③设在 msg3 末尾 → prefix 匹配第2次的写入 → 命中！只算 msg3
无缓存
100%
每轮重算全部内容
有缓存
≈10%
每轮只算新增消息

每轮新增的 1~2 条消息是唯一需要付费计算的部分，节省约 90% 费用。

什么时候才会撞到 20-block 天花板

只有一种极端场景：单轮对话内部产生了大量 block，多到超出了 20 的窗口。比如 assistant 在一轮回复里疯狂调用了 25 次 tool，中间没打任何断点。

正常情况：距离 ≪ 20，无需担心
第N轮: 断点在 block[15] → 写缓存 第N+1轮: 断点在 block[18] → 退3步找到 block[15] → 命中 第N+2轮: 断点在 block[21] → 退3步找到 block[18] → 命中
Claude Code 每轮都在最后一条消息放一个 cache_control，断点间距永远 ≪ 20。你不需要担心这个天花板。
一条请求的完整缓存命运

把前面所有机制串在一起，一次请求到达 Anthropic API 时的完整处理流程：

1
组装前缀：API 按 tools → system → messages 的顺序拼出完整 prompt 前缀
2
查找缓存：从请求中 cache_control 标记的最远位置开始，计算从 prompt 开头到当前位置的前缀哈希，去缓存库匹配；未命中就退一个 block，重新算更短前缀的哈希再试，最多退 20 步。这是一次循环查找，每次尝试的前缀终点都不同
3
分片计算：命中的前缀直接读缓存（节省 90%），未命中的尾部重新计算
4
写入缓存：每个断点位置各写入一条缓存（3 个断点 = 3 条嵌套条目，前缀长度依次递增）。命中的旧条目 TTL 自动刷新，未命中的写入新条目（按 1.25× 或 2× 计费）
5
TTL 管理：每条缓存独立计时，默认 5 分钟（可延长至 1 小时）。每被命中一次就免费刷新 TTL，超时未被命中则自动清理。BP① 和 BP② 几乎永不过期——每轮请求都会命中并刷新它们的 TTL 文档 ↗
两套模式，同一个底层

Anthropic 同时提供了两种缓存控制方式，底层缓存服务完全相同。文档 ↗

Explicit Breakpoints
Claude Code 当前路线

手动设置三个断点
精确控制每个断点位置和 TTL
适合需要将 tools 和 system
分开缓存的场景
Automatic Caching
Anthropic 后来推出的模式

顶层放一个 cache_control
API 自己决定断点位置
适合简单场景，减少心智负担
两种模式的底层缓存服务完全相同。区别在于谁来负责断点管理：automatic 交给 API，explicit 留在应用层。对于 Claude Code 这种需要将 tools 和 system 分开缓存（变更频率不同）的场景，手动控制是合理的。
缓存命中小技巧
1
别闲着超过 5 分钟。缓存 TTL 默认 5 分钟，超时后下一轮请求就得全量重算。保持对话节奏，每 2-3 分钟发一条消息，缓存一直活着。
2
大胆用 /rewind。回滚到之前的对话状态几乎零成本——那个前缀哈希早就存过了，直接命中。比删消息重发便宜得多。
3
别手贱改 system prompt。改一个字，级联失效一路炸到 messages。除非你真的需要调整指令，否则让 system 保持字节级一致。
🐶
不要停止对话。停下来喝杯水？可以。停下来吃个饭？你的缓存就没了。下次再聊，几十 KB 的 system prompt 从头算起——钱包在滴血。所以，别停，一直聊下去。 【狗头】
常见问题
Q: 我可以自己加第四个断点吗？
Anthropic 限制每个请求最多 4 个断点（explicit + automatic 合计）。Claude Code 用了 3 个，还有 1 个余量。但除非你有特殊的缓存分区需求，否则不需要——现有的三个断点位置已经能覆盖绝大多数场景。
Q: 如果我在中途改了 system prompt，之前 messages 的缓存还有效吗？
没有。system 的哈希变了，后面所有以前缀为基础的缓存全部失效，包括 messages。这就是级联失效。
Q: 1 小时 TTL 值得用吗？
写成本翻倍（1.25× → 2×，详见 1-hour cache duration），但如果你每隔 5~10 分钟回来继续同一段对话，1 小时 TTL 能让你回来时缓存还在。适合长时间的研究或调试会话。
Q: 不同的 Claude 模型共享缓存吗？
不共享。缓存跟模型版本绑定。Sonnet 的缓存不能给 Opus 用，反之亦然。
Q: prompt 里多一个空格缓存会失效吗？
会。缓存做的是精确前缀匹配——加密哈希，字节级别的。多一个空格、换一个大小写，整个哈希就变了，缓存全废。这也是为什么 system 和 tools 要保持字节级一致。
