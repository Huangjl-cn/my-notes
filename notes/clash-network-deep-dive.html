一切从一个迫在眉睫的需求开始

那天我需要下载 npm 包，速度十分缓慢。启动 Clash for Windows，打开界面那一刻——全局、规则、直连、脚本、TUN 模式、系统代理、TAP、虚拟网卡……密密麻麻的概念一股脑涌上来。

我开启了系统代理，终端似乎并没有成功代理，感觉很疑惑。这些模式到底是啥？规则是本地定的还是服务器定的？TUN 和系统代理有什么区别？

这些问题不搞清楚，出了岔子我连从哪下手排查都不知道。
规则模式：决策发生在哪？

直觉以为"规则是不是到了 VPN 服务器之后，由服务器来判断"。这个直觉是错的——四个模式全部是本地策略，在流量发出之前就已经决定了。

四种代理模式（全部本地决策）
直连 ──→ 所有流量直接发出，不走代理 全局 ──→ 所有流量都走代理节点 规则 ──→ 按规则集动态判断（国内直连 / 国外代理） 例：geoip:cn → 直连 DOMAIN-SUFFIX:google.com → 代理 脚本 ──→ 用 JS 自定义路由逻辑
全部在本地 Clash 内核中判断，与 VPN 服务器无关
系统代理：温和的中间人

系统代理修改操作系统的代理设置，告诉浏览器"先经过 Clash 的本地端口"。只对 HTTP/HTTPS 的应用生效，命令行工具、游戏、UWP 应用不走系统代理设置，就绕过去了。

PAC 脚本层
function FindProxyForURL(url, host) { if (dnsDomainIs(host, ".google.com")) return "PROXY 127.0.0.1:7890"; if (isInNet(host, "10.0.0.0", "255.0.0.0")) return "DIRECT"; return "PROXY 127.0.0.1:7890"; }
核心：HTTP 代理协议
浏览器 → TCP 连 127.0.0.1:7890 → 发送 CONNECT 指令： "CONNECT www.google.com:443 HTTP/1.1" → 告诉 Clash 我要连哪 → Clash 收到后自己发起连接 → 两边数据透传
系统代理本质：中间人转发。不是修改 DNS，而是告诉每个应用"你的网络请求交给本地这个端口，它来替你发"。只有遵守系统代理设置的应用才会走这个通道。
TUN 模式：网络层拦截

创建虚拟网卡，在操作系统层面拦截 所有 IP 数据包——不论浏览器的 HTTP、命令行的 TCP 还是系统更新的 UDP，只要发了 IP 包都会被截住。

系统代理TUN 模式
层级应用层网络层（IP 层）
原理修改系统代理设置创建虚拟网卡拦截所有包
接管范围仅 HTTP/HTTPS所有 TCP/UDP 流量
命令行 / 游戏不生效生效
性能开销轻量稍高
TUN ≠ 全局模式。TUN 解决的只是"哪些流量能进来"——所有 IP 包抓到 Clash 内核里后，按什么策略处理，还是由你选的模式（规则/全局/直连/脚本）来决定。TUN 只管"抓"，Clash 内核只管"判"。

Clash 的奇怪路由

Clash TUN 注册了一串看似古怪的路由——掩码不是标准的 /8、/16、/24，而是精心设计的一组覆盖：

八条路由拼起来覆盖了 1.0.0.0 ~ 255.255.255.255，几乎整个外网。

那为什么不直接写一条 0.0.0.0/0？两个原因：

避免冲突
  • 系统已有一条 0.0.0.0/0 → WLAN
  • 再加相同目标容易路由翻跳
  • 睡眠唤醒后可能丢失
  • 用不同目标避免冲突
破除死循环
  • Clash 自己连代理节点的包
  • 必须 绕过 TUN
  • 否则：Clash 发包 → TUN 拦截
  • → Clash 内核又处理 → 死循环
  • 多条路由更灵活做排除
接口 vs 网关：先选门，再选门外的人

多年以来对网络流向的理解："我的电脑 → 网关（路由器）→ 外网"。但只有一个网卡时不出问题，出现了多个虚拟网卡后，就不得不思考"从哪个接口出去"。

路由表解析：先选门，再选门外的人
包 → 查路由表匹配到 192.168.18.5/32 → 看接口列：用 10.8.0.233 这个网卡 ← 先选门 → 看网关列：把包发给 10.8.0.234 ← 再选门外的人

接口是网卡设备，IP 只是这个设备上的一个地址。设备上每个网卡有自己的 IP：

TUN 的暗度陈仓
Clash TUN 的路由
1.0.0.0/8 → 网关 198.18.0.2 → 接口 198.18.0.1

198.18.0.1 是你这一端的虚拟网卡，198.18.0.2 是"对端"。操作系统以为包是要发给另一个设备的，但 TUN 驱动实际上把包复制到了内存，直接塞给了本机上的 Clash 进程。

网关和接口都在同一台电脑上。 TCP/IP 协议栈设计上就必须有一个本端和对端地址，TUN 利用了这一点，假装有个远程网关，暗度陈仓把包转给了本地进程。
On-link：不是废话

10.90.23.0/24
  • WiFi 局域网内传文件、投屏
  • 通信目标在同一个二层网络
  • 不需要经路由器转发
198.18.0.0/16
  • Clash 的 fake-IP 段
  • 必须 On-link，否则系统会把包
  • 发给默认网关，Clash 截不到

判断逻辑：目标 IP & 子网掩码 == 本机 IP & 子网掩码 → 同一个子网 → 不需要网关 → On-link

Fake-IP：解决 DNS 污染

fake-IP 要解决的不是"包能不能被 Clash 抓到"，而是"Clash 怎么知道原始域名"：

没有 fake-IP
  • DNS → 真实 IP（可能已污染）
  • TUN 拦截 → 只有 IP
  • Clash：不知道对应什么域名
  • 只能走 geoip，精度低易误判
有 fake-IP
  • DNS → fake-IP（198.18.x.x）
  • TUN 拦截 → Clash 查映射表
  • Clash：198.18.1.1 = google.com
  • DOMAIN 规则完美生效
替代方案不可行：从 TLS 的 SNI 提取域名需要 TCP 重组，复杂昂贵，且 TLS 1.3 的 ECH 加密后提取不到。fake-IP 是更优雅的解决方案。

从浏览器到 Google 的完整链路

所有概念理清之后，用 curl 访问 Google 时，脑子里终于能清晰地浮现出整条链路：

1
浏览器 输入 URL，调 socket API → 连接请求(域名:443)
2
DNS Clash 返回 fake-IP → 目标 IP = 198.18.1.1
3
TCP 三次握手 → TCP 数据包
4
路由表 匹配 198.18.0.0/16 On-link → 走 Clash TUN 网卡
5
TUN 驱动 拦截 IP 包转到 Clash 进程
6
Clash 规则 查 fake-IP 映射表 → 选择日本节点
7
封装 加密原始包 → 构造发往代理节点的新包
8
WiFi 网卡 发出无线信号到路由器
9
路由器 NAT 转换 → 源 IP 换成公网 IP
10
代理节点 解密 payload → 取出原始包转发给 Google
11
Google 收到 "日本用户" 的请求 → 响应原路返回 ✅
包被穿了两次外套：
① TUN 虚拟网卡把包从"发往物理网络"改成"交给本地 Clash 进程"
② Clash 把原始包加密，外面套一个"发往代理节点"的新包

Google 从头到尾看到的源 IP 都是代理节点的 IP，既不知道你家宽带的公网 IP，也不知道你的局域网私有 IP。
