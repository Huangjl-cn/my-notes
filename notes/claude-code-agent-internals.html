缘起：一条命令引发的追问

在聊天框输入 /java-lsp，Claude Code 对 Java 文件执行了 LSP hover，返回了 diagnostics。但项目明明加了 Lombok，为什么 getter 方法的错误赫然在列？

检查 .lsp.json 发现 Lombok agent 的 -javaagent 参数已配置，只是 JDT LS 初始化需要时间。二次重试后，Lombok 生成的 getter 被正确解析。

一个正常的工具使用流程结束了。但问题才刚开始——LSP({...}) 到底是怎么工作的？
LSP 不是 shell 命令

一个常见误解——以为 LSP({...}) 是执行某个 shell 命令来调用 LSP。实际上，两者分工清晰：

.lsp.json
  • 配置 JDT LS 如何启动
  • javaagent、classpath 等参数
  • "怎么启动"
LSP({...}) 工具
  • 通过 LSP 协议与已启动的进程通信
  • 路径转 URI、行号自动转换
  • "怎么通信"
LSP 调用的底层映射

LSP({...}) 是 LLM ↔ LSP Server 之间的高层封装。参数会被转换成 JSON-RPC 请求：

你写的调用
LSP({ "operation": "hover", "filePath": "src/ResultUtils.java", "line": 49, "character": 25 })
发给 JDT LS 的请求
{ "jsonrpc": "2.0", "id": 1, "method": "textDocument/hover", "params": { "textDocument": { "uri": "file:///D:.../ResultUtils.java" }, "position": { "line": 48, ← 0-based "character": 24 ← 已转换 } } }
关键认知：LSP 处理了路径转 URI、行号 0-based/1-based 转换、方法名映射。它不是一个 shell 命令，而是翻译层。
Tool Use 机制

Claude Code 识别 LSP({...}) 不是靠"字符串模式匹配"，而是通过 Anthropic Messages API 的 Function Calling / Tool Use 能力——LLM 返回结构化 JSON：

LLM 返回消息中的 tool_use
{ "role": "assistant", "content": [ { "type": "text", "text": "我先看看..." }, { "type": "tool_use", ← 核心标记 "name": "LSP", ← 工具名 "input": { ← 参数 "operation": "hover", "filePath": "..." } } ] }

Claude Code 运行时看到 type: "tool_use"，执行流程如下：

1
路由 — 根据 name 找到工具实现（Read、Bash、LSP 等）
2
校验 — input 是否匹配工具定义的 JSON Schema
3
执行 — 读文件、跑命令、发 LSP 请求
4
返回 — 包装成 tool_result 喂回 LLM
核心认知：LSP({...}) 与 Read({...})、Bash({...}) 没有本质区别——都是 LLM 通过 Tool Use 机制调用的内置工具。
厂商格式：事实标准

这是 de facto standard（事实标准），不是 de jure standard（法定标准）。各家训练出来的格式各不相同：

概念 Anthropic OpenAI Gemini
工具定义 tools tools tools
调用标记 type: "tool_use" tool_calls 顶层字段 functionCall 在 parts 里
结果返回 type: "tool_result" role: "tool" role: "function"
底层逻辑完全一致：LLM 输出一段结构化 JSON → 运行时识别 → 执行 → 塞回结果到对话。
MCP：真正的开放协议

MCP（Model Context Protocol） 是 Anthropic 主导的开放协议，有公开规范、任何人可实现。所有消息基于 JSON-RPC 2.0，传输层可选 stdio、SSE 或 HTTP。

MCP 消息基础（JSON-RPC 2.0）
{ "jsonrpc": "2.0", "id": 1, "method": "...", "params": {} }
硬性要求：必须遵守的格式

MCP 对以下字段有强制规定，实现方不能省略或改名：

必须字段（硬性）
  • JSON-RPC 外壳： jsonrpc: "2.0"、id、method 三个字段固定
  • initialize 握手： 连接建立后必须先发 initialize 交换协议版本和能力
  • tools/list 响应： 每个工具必须有 name 和 inputSchema
  • tools/call 请求： 必须包含 name + arguments
  • tools/call 响应： 必须返回 content 数组，每项须有 type
可自定义（灵活）
  • 工具名与描述： name、description 你随便定
  • 参数结构： inputSchema 的 properties、required 全是你的领域逻辑
  • 传输层： stdio（子进程）、SSE（服务端推送）、HTTP 任选
  • 服务能力： 除 tools 外可选加 resources、prompts 等
  • 认证方式： 协议自身不限制，应用层自定
一句话： MCP 规定了"怎么包装请求/响应"的框架，但"工具叫什么、参数怎么传、数据怎么跑"全由你定。
协议三阶段
初始化
所有连接以 initialize 握手开始，双方交换能力信息和协议版本
注册
tools/list 响应中声明工具名、描述、inputSchema
调用
tools/call 传入 name + arguments，返回 content 数组
工具注册与调用的完整消息

MCP 的所有消息都套在 JSON-RPC 2.0 信封里。关键区分：

← 请求：method + params
{ "jsonrpc": "2.0", ← 固定 "id": 1, ← 请求 ID "method": "tools/call", ← 方法名 "params": { ← 参数放这里 "name": "search", ← 工具名 "arguments": { ← 工具参数 "query": "hello" } } }
→ 响应（或请求）：result
{ "jsonrpc": "2.0", ← 固定 "id": 1, ← 对应请求 ID "result": { ← 结果放这里 "content": [ { "type": "text", "text": "结果..." } ], "isError": false } }

服务端注册工具时，工具定义列表也放在 result 里返回：

tools/list 响应（也是 JSON-RPC 信封）
{ "jsonrpc": "2.0", "id": 1, "result": { "tools": [ { "name": "search", ← 必填：工具唯一名 "description": "搜索功能", ← 推荐：LLM 靠它理解用途 "inputSchema": { ← 必填：参数定义 "type": "object", "properties": { "query": { "type": "string" } }, "required": ["query"] } } ] } }
记住：响应必须有 result（成功时）或 error（失败时），二选一不能缺。
请求用 method + params 告诉服务端"调哪个工具、传什么参"；
响应用 result.content 返回执行结果，用 result.isError 标记工具自身异常。
最外层的 jsonrpc + id 是 JSON-RPC 2.0 协议外壳，每条消息都必须有。
MCP 优缺点
优势
  • 统一接口： 写一套 Server，所有 MCP Client 都能用，不用为每个 LLM 单独适配
  • 解耦职责： 工具开发者只需关注业务逻辑，不需要了解 LLM 内部机制
  • 实时推送： SSE 传输支持服务端主动推送结果，适合流式场景
  • 渐进增强： 可从纯 tools 起步，按需加 resources、prompts、sampling
  • 传输灵活： 本地 stdio（零配置）、远程 SSE（实时）、HTTP 按需切换
局限
  • 全量加载，不能惰性： tools/list 一次返回所有工具定义，全部塞进 LLM 上下文。挂 10 个 Server 每个 20 个工具，就是 200 个定义占着 token
  • 上下文污染： 不像 Claude Code 的 skill 按需触发、用完即走，MCP 的工具定义一直待在 system prompt 里，没用也在
  • 无标准认证： 协议不规定 auth 方式，公网服务得自己实现 token/OAuth
  • 调试门槛： 不像 REST 有 curl 可直接测，需要专用 Client 或 MCP Inspector
所以现在 CLI 工具反而更香： skill 只有被调用时才出现在上下文中，MCP Server 的工具定义则是永久占位。挂的 MCP Server 越多，浪费的 token 越多。MCP 适合稳定的、常用的工具集；偶发任务用 skill/CLI 按需触发更省上下文。
厂商调用 vs MCP
厂商工具调用 MCP
范围 单次 API 请求的响应格式 完整的应用层协议
标准化 厂商各自风格 公开规范，任何人可实现
实现方 LLM API 提供商 各种工具开发者
传输 单次 HTTP stdio、SSE、HTTP
串起来理解：tool_use / tool_calls 是 LLM 和 API 之间的约定；MCP 是 Claude Code 和外部工具 Server 之间的标准协议。
LLM 消息格式趋同

虽然各家字段名有差异，但 messages 数组 + role 区分 已是业界共识：

通用消息结构
{ "messages": [ { "role": "system", "content": "你是助手..." }, { "role": "user", "content": "看一下这个文件" }, { "role": "assistant", "content": "好的" }, { "role": "user", "content": "再帮我改一下" } ] }
各家消息差异
Anthropic OpenAI Gemini
消息字段 messages messages contents
系统消息 role: "system" role: "system" system_instruction 顶层
用户消息 role: "user" role: "user" role: "user"
助手消息 role: "assistant" role: "assistant" role: "model"
工具结果 role: "user" + type: "tool_result" role: "tool" role: "function"
多模态 content 数组 content 数组 parts 数组
核心规律：system / user / assistant 三个角色是标配。框架们（LangChain、Vercel AI SDK 等）都在这个模式上做抽象。
完整的 Agent 循环

从用户一句提问到最终回答，消息层面的完整流程：

Step 1：用户输入
POST /v1/messages { "model": "claude-sonnet-4", "system": "你是 Claude Code...", "messages": [ { "role": "user", "content":"看一下 ResultUtils.java 的内容..." } ], "tools": [ { "name": "Read", "input_schema": {...} } ] }
↓ tool_use
Step 2：LLM 决定调工具
{ "role": "assistant", "content": [ { "type":"text", "text":"我先读取这个文件" }, { "type":"tool_use", "id":"call_123", "name":"Read", "input":{"file_path":"src/ResultUtils.java"} } ] }
↓ tool_result 返回 → LLM 再次推理 → 继续调工具 → ... → 最终回答
完整的消息循环序列：
User → LLM: "帮我做X"  ← 初始提问
LLM → 运行时: "调用 Read 工具"  ← tool_use
运行时 → LLM: "Read 结果"  ← tool_result
LLM → 运行时: "调用 LSP 工具"  ← tool_use
运行时 → LLM: "LSP 结果"  ← tool_result
LLM → User: "最终回答"  ← 纯 text
Context Window 硬约束

每次 API 请求都携带完整的 messages 数组。因为 LLM 是无状态的 HTTP 服务——你得把"已经发生了什么"全部告诉它：

每次请求都完整发送
messages: [ { role: "user", content: "看一下 ResultUtils..." }, ← 原始提问 { role: "assistant", content: [ { type:"tool_use",... } ] },← LLM说"读文件" { role: "user", content: [ { type:"tool_result",... } ] },← 返回内容 { role: "assistant", content: [ { type:"tool_use",... } ] },← LLM说"调LSP" { role: "user", content: [ { type:"tool_result",... } ] },← 返回结果 // ↑ 以上每次都完整发送 ]
这就是 Context Window 是硬约束的根本原因——一次复杂任务调十几个工具，messages 数组轻松塞进几万甚至几十万 token。
message 管理 = Agent 优化核心

Agent 框架的优化本质都可以归结为 message list 管理：

优化 本质 效果
上下文压缩 多轮 tool_use/tool_result 浓缩成摘要 token 从几万降到几百
滑动窗口 丢弃最早消息，保留最近 N 轮 控制总长度
关键信息提取 只保留 tool_result 的核心结论 减少噪音
多轮汇总 多次类似提问合并 精炼历史
Agent 框架的工程天花板就是 message 管理——既要保证 LLM 有足够上下文做对事，又不能溢出 token 上限。
Prompt Caching：前缀即缓存

最后一层理解：Prompt Caching 基于前缀缓存，命中按开头算。 所以消息结构的排列有讲究：

消息结构排列
请求体: ├── system ← 系统提示词（不变） → ✅ 缓存命中 ├── tools ← 工具定义（不变） → ✅ 缓存命中 └── messages ← 对话历史（不断增长） → ❌ 每次都变 ├── user ← 原始提问 ├── assistant ← tool_use ├── user ← tool_result ├── assistant ← tool_use └── user ← tool_result

system 在最前面 + tools 紧随其后——只要不变，每次请求前几十 K token 直接从缓存读，只用计算末尾新增的 messages。如果 tools 定义每次都变，缓存就废了。

这也是为什么 Anthropic 建议把工具定义写得稳定、少用动态生成的 tools。
