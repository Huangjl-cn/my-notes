一条命令引发的追问

在聊天框输入 /java-lsp，Claude Code 对 Java 文件执行了 LSP hover，返回了 diagnostics。但项目明明加了 Lombok，为什么 getter 方法的错误赫然在列？

检查 .lsp.json 发现 Lombok agent 的 -javaagent 参数已配置——只是 JDT LS 初始化需要时间。二次重试后，Lombok 生成的 getter 被正确解析，一切恢复正常。

一个正常的工具使用流程结束了。但这次排查突然引发了我的思考：LSP({...}) 到底是怎么工作的？它看起来像一个函数调用——但谁在执行它？参数怎么传过去的？和 Read({...})、Bash({...}) 这些工具是同一类东西吗？

第一个认知：LSP 不是 shell 命令

一个常见的直觉——以为 LSP({...}) 背后是某个 shell 命令在启动 LSP 进程。实际上，两者分工清晰：

.lsp.json — 配置"怎么启动"
  • 声明 JDT LS 如何启动（javaagent、classpath 等参数）
  • 定义 workspace 路径、JDK 版本等环境配置
  • Claude Code 启动时读取，维护一个长期运行的 LS 进程
LSP({...}) 工具 — 配置"怎么通信"
  • 通过 LSP 协议与已启动的 LS 进程通信
  • 将人类友好的参数（路径、行号）转换成 JSON-RPC 请求
  • 每次调用是一次请求-响应，不涉及进程启停

这个区分很重要：.lsp.json 管的是 LS 进程的生命周期，LSP({...}) 管的是和这个进程的对话协议。理解了这一点，下一个问题自然浮现：LLM 是怎么"调用"一个工具的？

LSP 调用的底层映射

LSP({...}) 是 LLM ↔ LSP Server 之间的高层封装，参数会被翻译成标准 JSON-RPC 请求：

工具调用写法
LSP({ "operation": "hover", "filePath": "src/ResultUtils.java", "line": 49, "character": 25 })
发给 JDT LS 的请求
{ "jsonrpc": "2.0", "id": 1, "method": "textDocument/hover", "params": { "textDocument": { "uri": "file:///D:.../ResultUtils.java" }, "position": { "line": 48, ← 0-based "character": 24 ← 已转换 } } }

这个翻译层做了三件事：路径转 URI（Windows 盘符 → file:/// 格式）、行号 0-based 转换（编辑器行号从 1 开始，LSP 协议从 0 开始）、方法名映射（hover → textDocument/hover）。

LSP 不是 shell 命令——它是翻译层。LLM 不需要知道 LSP 协议的细节，它只需要表达"我要 hover 第 49 行"的意图。
Tool Use：LLM 如何表达调用意图

翻译层解决的是"怎么执行"，但在此之前还有一个更基础的问题：LLM 怎么表达"我想调工具"？

这不是靠字符串模式匹配。Claude Code 使用的是 Anthropic Messages API 的原生 Tool Use 能力——LLM 直接返回结构化 JSON，其中内嵌 type: "tool_use" 标记：

LLM 返回消息（简化的 content block）
{ "role": "assistant", "content": [ { "type": "text", "text": "我先看看这个文件的内容..." }, { "type": "tool_use", ← 核心标记：这不是文本，是一个工具调用 "id": "call_abc123", ← 本次调用的唯一 ID，用于匹配结果 "name": "Read", ← 工具名 "input": { ← 传给工具的参数，JSON Schema 校验 "file_path": "src/ResultUtils.java" } } ] }

content 是一个数组——同一条 assistant 消息里可以同时包含文本解释和工具调用。LLM 可以边说话边动手。

执行四步：从 LLM 决定到结果回传

Claude Code 运行时收到 type: "tool_use" 后的处理流程：

1
路由 — 根据 name 字段找到对应的工具实现。Read 映射到文件系统读取，Bash 映射到 shell 执行，LSP 映射到 Language Server 通信——但 LLM 不关心这个映射，它只知道工具名
2
校验 — 将 input 对工具定义的 JSON Schema 做校验。如果 LLM 传了不存在的字段或缺少必填参数，运行时在这一步拦截，不给工具执行的机会
3
执行 — 校验通过后实际执行：读文件、跑 shell 命令、发 LSP 请求。这一步是唯一有副作用的环节
4
回传 — 将执行结果包装成 type: "tool_result"，以 role: "user" 的身份追加到消息列表末尾，喂回 LLM 做下一轮推理
LSP({...}) 和 Read({...})、Bash({...}) 在 LLM 眼里没有本质区别——都是 Tool Use 机制下的一个 name。工具实现可以千差万别，但 LLM 只看到统一的结构化调用接口。
厂商格式：事实标准的分化

Tool Use 是 de facto standard（事实标准）——各家 LLM 厂商都支持，但具体格式是各自设计的。没有 ISO 规范，只有"大家都这么干"：

概念AnthropicOpenAIGemini
工具定义tools 数组tools 数组tools 数组
调用标记type: "tool_use"tool_calls 顶层字段functionCall 在 parts 里
参数位置input 字段（JSON 对象）function.arguments（JSON 字符串）args 字段（JSON 对象）
结果返回type: "tool_result"role: "tool"role: "function"
并行调用content 数组中多个 tool_usetool_calls 数组中多项parts 中多个 functionCall
底层逻辑完全一致，但字段名、嵌套结构、参数编码方式各不相同。直接对接各家 API 的话，需要为每个厂商写一套解析逻辑。
为什么没有统一标准？

LLM 的 Tool Use 能力是 2023–2024 年各家独立演进的产物。Anthropic 在 Claude 2 时率先引入 First-Party Tool Use（content block 内嵌 tool_use），OpenAI 在 GPT-4 发布时采用了不同的 tool_calls 顶层字段方案，Google 则在 Gemini 的 parts 结构中承载 functionCall。

没有标准委员会先于实现——每家都在 API 设计上做了不同取舍：Anthropic 选择把 tool_use 和 text 平级放在 content 数组中（允许交错思考），OpenAI 选择把 tool_calls 作为 assistant message 的独立顶层字段（和 content 并列），Google 则因为 Gemini 本身就是多模态 parts 架构，functionCall 自然成了 parts 的一种类型。

直到 Anthropic 提出 Model Context Protocol (MCP)，才有了第一个跨厂商的工具通信标准尝试——详见下一 Tab。

Tool Use 的本质

抛开所有术语，Tool Use 就是三步：

LLM
输出一段结构化的 JSON，声明"我要调用 X 工具，参数是 Y"
Runtime
识别 tool_use 标记，路由到对应实现，执行后将结果包装成 tool_result
Loop
结果追加到消息历史，LLM 再次推理——可能继续调工具，也可能给出最终回答

这个循环的工程上限，就是消息历史的管理能力——后面再展开。先看一个关键进展：MCP 如何把厂商私有格式变成了开放标准。

MCP：给工具通信定一个开放标准

Model Context Protocol (MCP) 是 Anthropic 主导的开放协议。设计动机很直接：Tool Use 在各家 API 里的格式各不相同——MCP 要解决的是工具实现方和 LLM 平台方的解耦问题。

MCP 选择了 JSON-RPC 2.0 作为消息信封——这是一个成熟、简单、语言无关的远程调用协议：

MCP 消息基础（所有消息都套在这个外壳里）
{ "jsonrpc": "2.0", "id": 1, "method": "...", "params": {} }

传输层可选 stdio（子进程标准输入输出，零网络配置）、SSE（Server-Sent Events，适合服务端推送场景）或 HTTP。本地工具用 stdio 最直接——Claude Code 启动一个 MCP Server 子进程，通过 stdin/stdout 交换 JSON-RPC 消息。

协议的骨架：什么必须遵守、什么可以自由

MCP 的设计哲学是规定通信框架，不规定业务逻辑：

硬性要求（不遵守就无法互通）
  • JSON-RPC 外壳：每条消息必须有 jsonrpc: "2.0"、id、method（请求）或 result/error（响应）
  • initialize 握手：连接建立后必须先交换 initialize 请求/响应，协商协议版本和双方能力
  • tools/list 响应结构：每个工具必须有 name（唯一标识）和 inputSchema（JSON Schema 定义参数）
  • tools/call 请求/响应：请求必须含 name + arguments；响应必须返回 content 数组，每项含 type
自由空间（完全由实现方决定）
  • 工具名与描述：name 和 description 由实现方定义——LLM 靠 description 理解何时使用
  • 参数结构：inputSchema 里的 properties、required 完全是领域逻辑
  • 传输层：stdio（子进程）、SSE（推送）、HTTP 任选，协议层不依赖传输
  • 扩展能力：除 tools 外，可选实现 resources（暴露数据）、prompts（预置提示模板）、sampling（服务端发起 LLM 请求）
  • 认证方式：协议自身不限制，应用层自行决定 token/OAuth/互相 TLS
MCP 规定了消息怎么封包、怎么路由，但"工具有什么能力、参数长什么样、数据怎么处理"是实现方的事。框架统一，内容自由。
协议三阶段：初始化 → 发现 → 调用

每个 MCP 连接的生命周期分三个阶段，顺序固定：

初始化
Client 发 initialize 声明自己的协议版本和能力；Server 回应支持的版本和自身能力集。只有握手成功，后续通信才被接受
发现
Client 发 tools/list 请求，Server 返回所有可用工具的定义列表（name + description + inputSchema）。LLM 根据这些定义决定何时调用哪个工具
调用
Client 发 tools/call 传入工具名和参数；Server 执行后返回 content 数组。一次连接可发起多次调用，无需重复握手
一条完整的工具调用消息

把三个阶段的消息串起来看——这是一个完整 MCP 工具调用的请求和响应：

← 请求：告诉 Server 调哪个工具
{ "jsonrpc": "2.0", ← JSON-RPC 外壳，固定 "id": 7, ← 请求 ID，响应时原样返回 "method": "tools/call", ← MCP 方法名 "params": { ← 参数放在这里 "name": "search", ← 工具名 "arguments": { ← 工具的业务参数 "query": "context window optimization" } } }
→ 响应：Server 返回执行结果
{ "jsonrpc": "2.0", ← 外壳一致 "id": 7, ← 对应请求 ID "result": { ← 结果放在这里 "content": [ ← 必须是一个数组 { "type": "text", "text": "找到 3 篇相关文档：\n1. ..." } ], "isError": false ← 标记工具自身是否出错 } }

工具注册发生在初始化之后——Server 通过 tools/list 响应告诉 Client 自己有哪些工具：

tools/list 响应（也是 JSON-RPC 信封）
{ "jsonrpc": "2.0", "id": 2, "result": { "tools": [ { "name": "search", ← 必填：工具唯一标识名 "description": "搜索知识库中的文档", ← 推荐：LLM 根据这段文字判断何时使用 "inputSchema": { ← 必填：参数定义 "type": "object", "properties": { "query": { "type": "string", "description": "搜索关键词" }, "maxResults": { "type": "number", "default": 10 } }, "required": ["query"] } } ] } }
① 请求用 method + params 表达"调哪个工具、传什么参"
② 响应用 result.content 数组返回执行结果，result.isError 标记工具自身异常
③ 最外层的 jsonrpc + id 是 JSON-RPC 2.0 协议外壳——每条消息都必须有
MCP 的取舍：统一接口的代价
优势
  • 统一接口：写一套 MCP Server，所有 MCP Client 都能用——不用为每个 LLM 平台单独适配工具
  • 解耦职责：工具开发者只需关注业务逻辑，不需要了解各家 LLM API 的 tool_use / tool_calls 格式差异
  • 实时推送：SSE 传输支持服务端主动推送结果，适合流式、长时间任务
  • 渐进增强：从纯 tools 起步，按需加 resources（暴露数据源）、prompts（预置提示）、sampling（Server 请求 LLM）
  • 传输灵活：本地 stdio（零配置调试）、远程 SSE（实时推送）、HTTP 按场景切换
局限
  • 全量加载，不能惰性：tools/list 一次性返回所有工具定义，全部塞进 LLM 上下文。挂 10 个 MCP Server 每个 20 个工具 = 200 个工具定义常驻 system prompt
  • 上下文永久占用：MCP 的工具定义一直待在 system prompt 里，即使这次对话根本用不到。和 Claude Code 的 skill（按需触发、用完即走）是完全不同的哲学
  • 无标准认证：协议不规定 auth 方式——公网 MCP Server 需要自行实现 token / OAuth / mTLS
  • 调试门槛：不像 REST API 可以用 curl 直接测——需要 MCP Inspector 或专用 Client
MCP vs Skill/CLI：上下文成本的两种路线

MCP 和 Claude Code Skill 代表了工具集成的两种根本策略：

维度MCP ServerSkill / CLI 工具
工具定义位置常驻 system prompt只在被调用时注入 context
上下文成本固定开销（N 个工具 × 每个定义 ~200 token）零开销直到触发
适用场景高频、稳定、通用工具（搜索、数据库、文件系统）偶发、领域特定、复杂工作流
发现机制LLM 从 tool list 中选择用户显式调用或用 description 匹配
工具数量上限受 context window 约束，建议 < 50 个理论上无限，只有调用的才占用 token

MCP 和 Skill/CLI 不是二选一的问题。高频核心工具放 MCP（稳定、通用、值得占用上下文），偶发复杂工作流用 Skill/CLI 按需触发（不调用就不花钱）。判断标准很简单：这个工具在多高比例的对话中会被用到？

消息格式的高度趋同

Tool Use 和 MCP 解决的是"怎么调用"，但它们都依赖一个更基础的数据结构：消息数组。不管哪家 API，核心模式都是 messages 数组 + role 区分发言者：

通用消息结构（伪代码，所有厂商都是这个模式）
{ "messages": [ { "role": "system", "content": "你是 Claude Code，一个 AI 编程助手..." }, { "role": "user", "content": "看一下 ResultUtils.java 的内容" }, { "role": "assistant", "content": "好的，我来读取这个文件" }, { "role": "user", "content": "再帮我检查 LSP diagnostics" } ] }

但`role` 的命名和消息承载方式在不同厂商之间有微妙差异：

消息数组messagesmessagescontents
系统消息顶层 system 字段（不在 messages 里）role: "system"system_instruction 顶层字段
用户消息role: "user"role: "user"role: "user"
助手消息role: "assistant"role: "assistant"role: "model"
工具结果role: "user" + content 中 type: "tool_result"role: "tool" + tool_call_idrole: "function"

system / user / assistant 三个角色是所有厂商的标配。框架层（LangChain、Vercel AI SDK、LlamaIndex）都在这个模式上做抽象——统一 messages 格式，适配不同 API。
完整的 Agent 循环：从提问到回答的消息序列

消息格式清楚了，接着看一个完整的 Agent 交互循环。从用户一句"帮我看看这个文件"到最终回答，消息层面经历了多轮往返：

Step 1：用户输入 → API 请求
POST /v1/messages { "model": "claude-sonnet-4-20250514", "system": "你是 Claude Code，一个 AI 编程助手...", "messages": [ { "role": "user", "content": "看一下 ResultUtils.java 的内容..." } ], "tools": [ { "name": "Read", "input_schema": {...} }, { "name": "Bash", "input_schema": {...} }, { "name": "LSP", "input_schema": {...} } ] }
↓ LLM 返回 tool_use
Step 2：LLM 决定调用 Read
{ "role": "assistant", "content": [ { "type": "text", "text": "我先读取这个文件" }, { "type": "tool_use", "id": "call_123", "name": "Read", "input": { "file_path": "src/ResultUtils.java" } } ] }
↓ tool_result 返回 → LLM 再次推理 → 可能继续调工具 → ... → 最终 text 回答
从消息视角看完整的一次 Agent 任务：
① User → LLM: "帮我检查代码问题"  ← 初始提问，只有一条 user message
② LLM → Runtime: tool_use(name="Read")  ← LLM 决定先读文件
③ Runtime → LLM: tool_result(content="文件内容...")  ← 把文件内容喂回去
④ LLM → Runtime: tool_use(name="LSP", operation="diagnostics")  ← LLM 读了代码后决定查 LSP
⑤ Runtime → LLM: tool_result(content="3 errors, 1 warning...")  ← 返回诊断结果
⑥ LLM → User: "发现 3 个问题：..."  ← 综合分析后给出最终回答
Context Window：为什么消息数组是硬约束

这个循环有一个关键前提：LLM 是无状态的 HTTP 服务——它不记得上一轮说了什么。每次 API 请求，必须把整个消息历史完整发送：

第 N 轮请求——下面这整块都是本次 HTTP body
messages: [ { role: "user", content: "看一下 ResultUtils..." }, ← 原始提问（第 1 轮） { role: "assistant", content: [ ← LLM 决定读文件（第 2 轮） { type: "tool_use", name: "Read", ... } ]}, { role: "user", content: [ ← 文件内容回传（第 3 轮） { type: "tool_result", content: "package com...\n..." } ]}, { role: "assistant", content: [ ← LLM 决定查 LSP（第 4 轮） { type: "tool_use", name: "LSP", ... } ]}, { role: "user", content: [ ← LSP 结果回传（第 5 轮） { type: "tool_result", content: "3 errors found..." } ]}, // ↑ 以上所有消息，每次请求都要完整发送 // ↓ 下面是本轮新增的 { role: "assistant", content: "分析结果如下：..." } ]

一次复杂任务调十几个工具、读几个大文件——messages 数组轻松塞进几万甚至几十万 token。Context Window 不是"建议"，是硬约束。

为什么 LLM API 设计成无状态的？

这是工程权衡的结果。有状态方案（服务端记住对话历史）看似节省带宽，但带来更严重的问题：服务端需要维护海量会话状态 → 单点故障丢失所有对话 → 无法水平扩展 → 计费和速率限制变得极其复杂。

无状态方案让每一次 API 请求都是自包含的：网关可以任意路由到空闲 GPU、请求之间完全独立、失败只需重发当前请求。代价是每次都要把历史重传一遍——但这个代价换来了系统的水平扩展能力和容错性。

token 优化的重要性正源于此。无状态架构意味着每多一个冗余 token，不是浪费一次——是浪费每一轮。

消息历史是 Agent 的命脉

回过头看，Agent 的能力上限不是模型有多聪明——是消息历史能装多少有效信息。tool_use 可以做任何事，但每次调用的结果都在挤占 Context Window。Message 管理就是 Agent 优化的核心战场。

Message 管理 = Agent 优化的核心

每次请求都要带完整历史，怎么管理这个消息数组就成了 Agent 框架的工程天花板。四种经典策略：

策略做法效果代价
上下文压缩 将多轮 tool_use / tool_result 浓缩为一段摘要，替代原始消息 token 从几万降到几百 细节丢失——LLM 可能漏掉关键信息
滑动窗口 丢弃最早的消息，只保留最近 N 轮 总长度可控 丢失早期上下文——Agent 可能"忘记"最初的目标
关键信息提取 从 tool_result 中只保留核心结论，丢弃原始输出 减少噪音，提升信号密度 需要准确的提取逻辑——提取错误比不提取更糟
分层记忆 近期消息保留完整、中期保留摘要、远期只保留关键事实 平衡细节与容量 实现复杂度最高，三层管理逻辑需要精心调优
四种策略在不同场景下各有优劣，没有哪种能通吃所有情况。工程上的判断标准是：当前任务需要多少历史信息才能做出正确决策？少了会出错，多了浪费 token——这就是 Agent 优化的核心矛盾。
Prompt Caching：前缀决定一切

除了精简 messages，另一条路线思路完全不同：不减少发送量，但让重复部分免费。

Anthropic 的 Prompt Caching 机制的核心原理是前缀缓存——如果两次请求的前 N 个 token 完全相同，第二次请求的前 N 个 token 直接从缓存读取，只计算新增部分。这意味着消息在请求体中的排列顺序直接决定缓存命中率：

请求体结构 → 缓存行为
POST /v1/messages 请求体: ├── system ← 系统提示词（每轮不变） → ✅ 总是命中缓存 ├── tools ← 工具定义列表（每轮不变） → ✅ 总是命中缓存 └── messages ← 对话历史（每轮新增） → ❌ 末尾变化导致缓存断点 ├── user: "帮我检查代码" ← 第 1 轮 ├── assistant: tool_use(Read) ← 第 2 轮（新增，未缓存） ├── user: tool_result(文件内容) ← 第 3 轮（新增，未缓存） ├── assistant: tool_use(LSP) ← 第 4 轮（新增，未缓存） └── user: tool_result(诊断结果) ← 第 5 轮（新增，未缓存）

关键点：system 和 tools 必须放在 messages 前面——而且每次请求保持不变。如果 tools 定义每次动态生成，缓存就废了——前几十 K token 每次都要重新计算。

缓存命中率的工程细节

Prompt Caching 有几个非直觉的特性：

① 最小缓存长度：Anthropic 要求缓存块至少 1024 token（Claude Sonnet）或 2048 token（Claude Opus）。太短的 system prompt 反而不值得缓存。

② 缓存写入成本：写入缓存比正常推理贵——通常多 25% 的 token 计费。所以只有确定会重复使用的内容才值得写入缓存。一次性请求不应该标记缓存。

③ 断点即失效：缓存基于前缀——中间插入一条消息，后面的所有缓存全部作废。所以 tools 必须放在 messages 前面且保持稳定——如果 tools 定义中间插入了一个新工具，整个 messages 数组的缓存位点都会漂移。

④ 实际收益：在 Agent 场景中，system + tools 通常占 20-40K token。按 Anthropic 定价，缓存命中后这部分成本的折扣率可达 90%。对于高频 Agent 任务，这意味着每次工具调用的增量成本远低于首次请求。

⑤ Claude Code 的实践：Claude Code 将 system prompt 和 tools 定义设计为固定前缀——包括工具名、描述、JSON Schema 都尽量不动态变化。这也解释了为什么 Claude Code 的工具定义非常稳定、很少动态生成 tools：稳定 = 缓存友好 = 省钱。

缓存友好的架构设计

Prompt Caching 的原理反过来影响架构决策：

1
固定前缀优先：所有不变内容（system、tools、few-shot examples）放在请求体最前面，确保它们在缓存前缀范围内
2
动态内容后置：新增的 user message、tool_result 只能追加在末尾——不要插入中间，避免破坏后续缓存位点
3
稳定工具定义：不要每次请求动态生成 tools JSON Schema——用固定的工具配置文件，保证缓存命中
4
Minimal context：只有确实会被多次调用的内容才写缓存——一次性 tool_result 不标记 cache control
缓存友好的架构就是让"不变的"呆在前面、让"变化的"追加到末尾、让"不确定是否复用的"别占缓存位。

几条核心结论
① 工具调用的本质是结构化 JSON，不是 shell 命令
LLM 输出 type: "tool_use" 的 JSON block → Runtime 识别、路由、执行 → 结果包装成 tool_result 回传。Read、Bash、LSP 在 LLM 眼里只是 name 不同——底层机制完全一样。
② 厂商格式趋同但不统一——MCP 补上了标准化这一环
Anthropic 的 tool_use、OpenAI 的 tool_calls、Gemini 的 functionCall 都表达同一件事，只是字段名和嵌套结构不同。MCP 在更高的抽象层（Runtime ↔ 工具 Server）提供了统一的 JSON-RPC 2.0 协议——一次实现，跨平台使用。
③ Agent 循环的工程上限 = messages 数组管理
每次 API 请求携带完整消息历史——LLM 不记得上一轮说了什么。tool_use / tool_result 每多一轮，messages 就胖一圈。Context Window 不是建议，是物理定律。
④ Prompt Caching 靠前缀——架构设计要主动适配
缓存命中按开头算。system + tools 放在请求体最前面且保持稳定 → 每次请求前几十 K token 直接从缓存读取。动态 tools 定义、中间插入消息 → 缓存失效。架构的稳定性 = 缓存的命中率 = 成本。
⑤ MCP 和 Skill 不是竞争关系——是频率分层
MCP 适合高频核心工具（稳定、通用、值得占上下文），Skill/CLI 适合偶发复杂工作流（按需触发、不调用零成本）。分界线是：这个工具在多高比例的对话中会被用到？
